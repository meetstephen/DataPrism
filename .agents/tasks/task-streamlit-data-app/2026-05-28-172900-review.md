# Streamlit multi-page data analysis application (greenfield)

A greenfield Streamlit app built from scratch: four pages (dashboard, upload-and-analyze, AI insights, advanced analytics), a synthetic data generator, reusable Plotly chart helpers, and an OpenAI integration with rule-based fallback. The architecture follows Streamlit's multi-page convention with shared state via `st.session_state` and utility modules in `utils/`. The app is designed for Streamlit Community Cloud deployment with no server-side persistence beyond session state.

**Watch for:** OpenAI API key is accepted via a UI text input and passed directly to the client constructor with no validation or sanitization (confirmed). The error path in `generate_insights_openai` exposes raw exception messages — including potentially sensitive API error details — back to the user (confirmed). Uploaded file processing has no size or content guards (confirmed).

## High-level view

The data generator produces a fixed 800-row synthetic dataset seeded at 42, with course assignment correctly coupled to major but professor assignment entirely random — professors teach any course in any department, which undermines the realism of any cross-analysis between professor and major/course. This is a design choice for a demo, but it limits how meaningful the pre-built dashboard insights actually are.

The AI insights module provides a clean two-path architecture: OpenAI for natural-language analysis when a key is available, rule-based heuristics otherwise. The fallback is thorough (variability checks, correlation surfacing, missing-data alerts) and will produce useful output on arbitrary datasets. However, the OpenAI path constructs its prompt from a stringified dict of summary stats rather than a curated data snapshot, which means the model receives Python repr noise instead of clean tabular context.

The upload-and-analyze page processes user-supplied files (CSV or Excel) with broad exception handling but no constraints on file size, row count, or column count. A multi-megabyte Excel upload could lock the Streamlit event loop during parsing because everything runs synchronously.

The advanced analytics page provides a flexible pivot-table and chart builder. The line-chart path uses a raw `px.line` import inside the button handler rather than routing through the visualization utilities, breaking the abstraction boundary the rest of the app maintains.

1. **OpenAI error message leaks to users** — `generate_insights_openai` catches all exceptions and returns `f"Error generating AI insights: {str(e)}"` directly into the UI. Return a generic user-facing message and log the details server-side.
2. **No upload size or content guards** — `2_Upload_and_Analyze.py` reads uploaded files with no limits. Add Streamlit's `max_upload_size` config or check `uploaded_file.size` before parsing.
3. **Prompt construction uses raw dict repr** — The AI path passes `summary_stats` as a Python dict stringification. Format it as a markdown table or structured text for better LLM comprehension.
4. **Inline plotly import breaks visualization abstraction** — `4_Advanced_Analytics.py` line-chart path imports `plotly.express` directly instead of using a `create_line_chart` utility, which means that chart won't pick up the shared template and styling.
5. **Duplicate `set_page_config` calls across pages** — Every page file calls `st.set_page_config()`, but Streamlit only respects the first call per session. If a user navigates between pages in certain deployment modes, these calls can raise warnings or be silently ignored.

## OpenAI integration and error exposure

The `generate_insights_openai` function wraps the entire OpenAI call in a bare `except Exception as e` and returns the exception's string representation directly into the Streamlit UI:

```python
except Exception as e:
    return f"Error generating AI insights: {str(e)}. Falling back to rule-based analysis."
```

OpenAI client exceptions can contain request IDs, partial API keys in URL fragments, rate-limit metadata, and org identifiers. Surfacing these in a web-facing app is an information-disclosure risk. The function also claims it's "falling back" but doesn't actually trigger the fallback — the caller in `3_AI_Insights_Engine.py` displays whatever string comes back and never invokes `generate_insights_fallback` on this path. So a failed API call produces an error message displayed as "AI insights" with no actual analysis.

The prompt itself is assembled by interpolating a Python dict's `__repr__` into the user message. The resulting string contains curly braces, square brackets, and Python type annotations that waste tokens and confuse the model. A structured markdown table of the top statistics would produce better outputs and cost fewer tokens.

## Upload pipeline — no resource bounds

`2_Upload_and_Analyze.py` accepts CSV and Excel files and immediately reads the entire file into a DataFrame:

```python
if uploaded_file.name.endswith(".csv"):
    df = pd.read_csv(uploaded_file)
else:
    df = pd.read_excel(uploaded_file)
```

There is no check on `uploaded_file.size`, no row-limit parameter, and no timeout. Streamlit's default `server.maxUploadSize` is 200 MB. A 200 MB CSV could easily produce a DataFrame that exhausts the container's memory on a free-tier Streamlit Cloud deployment (typically 1 GB RAM). Even a moderately large Excel file with many sheets and formulas can block the event loop for tens of seconds during `openpyxl` parsing.

Additionally, the broad `except Exception as e` at the bottom catches parse failures but also suppresses any unexpected error (e.g., `MemoryError`), displaying only a generic message.

## Visualization abstraction gap in Advanced Analytics

The custom chart builder in `4_Advanced_Analytics.py` handles five chart types. Four route through the shared `utils/visualizations.py` functions. The "Line" type does not:

```python
elif chart_type == "Line":
    ...
    import plotly.express as px
    fig = px.line(chart_df, x=x_axis, y=y_axis, color=color_param, title=...)
    fig.update_layout(template="plotly_dark", height=400)
```

This means the line chart misses the shared font, margin, legend-position, and color-sequence configuration that `_apply_layout` provides. The inline `import plotly.express as px` inside a conditional block is also unusual — it works, but it suggests the line chart was a late addition that skipped the utility layer.

## Data generator — professor-course decoupling

Professors are assigned to rows uniformly at random, independent of major or course. A nursing student taking "Clinical Practicum" might be listed under "Dr. Michael Chen" who also appears teaching "Algorithms" to a CS student. For a demo dataset this is cosmetically fine, but any analysis that tries to draw conclusions about professor effectiveness per course or department will produce noise rather than signal. Since the dashboard prominently features "Top Professors by Avg Evaluation" and "Professors by Avg Course Cost," the results these charts show are artifacts of random assignment rather than any meaningful pattern.

## Duplicate page configuration

Every page file calls `st.set_page_config(...)` with its own title and icon. In Streamlit's multi-page app architecture, only `app.py`'s config is guaranteed to take effect. The page-level calls work in current Streamlit versions (each page runs as its own script), but this relies on implementation behavior that has changed between versions. If the deployment target pins `streamlit>=1.28.0`, this is currently safe, but it creates fragility against future upgrades.
