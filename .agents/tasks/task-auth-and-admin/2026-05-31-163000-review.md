# Authentication gate, RBAC, and admin panel for DataPrism

This change adds a full authentication layer to DataPrism: sign-up/sign-in via Supabase Auth, role-based access control (admin > analyst > viewer), per-user data scoping in the database layer, an admin panel for user/activity/feedback management, and the supporting SQL schema in SUPABASE_SETUP.md. When Supabase is not configured, the app continues to run in local-dev mode with a mock admin user, preserving the zero-friction developer experience.

**Watch for:** RLS policies on `dp_users` and `dp_activity_log` allow unrestricted access to the `anon` role, which means any visitor with the anon key can read all user profiles and insert arbitrary activity log entries (confirmed). The admin panel's role-change mechanism fires on selectbox value change during every Streamlit rerun, risking unintentional role mutations (likely). Data-scoping in `database.py` relies solely on application-level filtering rather than RLS, so a direct Supabase call (or a bug in the filtering logic) would expose other users' data (confirmed).

## High-level view

The auth module (`utils/auth.py`) switches between "local dev" and "real auth" based on whether Supabase credentials exist. No other page needs to know about the dual-mode behavior, but the mock user writes `dp_authenticated = True` and `dp_user_role = "admin"` into session state on every run â€” silently granting admin if Supabase is misconfigured (URL present but key missing) rather than only when it's entirely absent.

Data scoping is enforced at the application layer in `database.py`'s `list_*` functions via a `.eq("user_id", user_id)` filter when the user is not an admin. The Supabase RLS policies do not enforce the same constraint, meaning a compromised or buggy client could access all rows.

The admin panel is gated by `require_role("admin")` and provides user listing, role changes, activity log viewing, and feedback display. The role-change widget uses a selectbox that triggers on any value mismatch from the current role, with no confirmation step.

The RLS policies documented in SUPABASE_SETUP.md grant the `anon` role full SELECT, INSERT, and UPDATE on `dp_users` and full INSERT + SELECT on `dp_activity_log`. Because the Supabase client is initialized with the anon key (by design, per `supabase_client.py`), these policies effectively disable row-level security for those tables.

## RLS policies grant anon full access to user data

The documented SQL creates three `anon`-role policies on `dp_users`:

```sql
create policy "dp_users_anon_insert" on public.dp_users
    for insert to anon with check (true);

create policy "dp_users_anon_select" on public.dp_users
    for select to anon using (true);

create policy "dp_users_anon_update" on public.dp_users
    for update to anon using (true) with check (true);
```

Because the app uses the anon key (not a service-role key), these policies mean any client-side REST call using the project's public anon key can enumerate all users, change anyone's role to admin, or deactivate accounts. This is a privilege-escalation vector: an attacker who knows the project URL and anon key (both are visible in a deployed Streamlit app's network traffic) can make themselves admin.

The same pattern appears on `dp_activity_log` â€” anon can insert anything and read everything. Combined, this means user enumeration, activity surveillance, and log injection are all possible without authentication.

The fix is either to use authenticated-only policies (the app signs in via `sign_in_with_password` and gets a JWT, so subsequent calls should use the `authenticated` role not `anon`) or to use the service-role key server-side for admin operations. The sign-up flow's insert into `dp_users` can be handled by the `handle_new_user` trigger (which already exists and runs as `security definer`), eliminating the need for anon insert.

## Application-level data scoping without RLS backup

`database.py` filters queries with `.eq("user_id", user_id)` when the user is not an admin, but the underlying tables have no RLS policies that enforce user isolation. The `user_id` columns are added via `ALTER TABLE ... ADD COLUMN IF NOT EXISTS user_id uuid` in the SQL guide, but no corresponding policies restrict SELECT/UPDATE/DELETE to rows where `user_id` matches `auth.uid()`.

Supabase exposes a direct REST API. Anyone with the anon key (or a valid JWT given the wide-open anon policies) can query `dp_datasets` without the `.eq("user_id", ...)` filter and get all rows. For a multi-tenant app, defense-in-depth requires RLS to match the application filter.

## Admin panel role-change fires without confirmation

In the Users tab, each user row renders a selectbox with the current role as default. When the selected value differs from `current_role`, the update fires immediately:

```python
if new_role != current_role:
    client.table("dp_users").update({"role": new_role}).eq("id", u["id"]).execute()
    st.rerun()
```

Streamlit reruns the entire page on any widget interaction. If an admin changes a filter in the activity tab and comes back, or if the page reruns for another reason, selectbox state might reset or drift. A separate "Save changes" button per user row, or a batch-save approach, would prevent accidental mutations.

## Silent fallback to mock admin on partial config

If Supabase credentials are partially configured (URL present but key missing, or key present but URL wrong), `is_configured()` returns `False` and the app silently falls back to mock admin mode with no warning. A user who thinks they configured auth is running without it. Adding a warning when one of `SUPABASE_URL` / `SUPABASE_KEY` is set but the other is missing would surface the misconfiguration.

`app.py` also logs a `"login"` event inside the `session_restored` block, which runs on every first page load â€” including when the user was already authenticated from a prior session. Combined with the login page's own `"login"` log on sign-in success, a single authentication produces two log entries, and a returning user whose session was restored (without signing in again) also produces a spurious entry.
