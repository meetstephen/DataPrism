# Analysis toolkit expansion: joins, SQL, transforms, templates, annotations, data dictionary

This change adds six features that round out DataPrism's analyst workflow: dataset joining/merging, a SQL query interface (via pandasql/SQLite), column transformations on the cleaning page, saved analysis templates persisted to Supabase or local files, in-session chart annotations, and an auto-generated data dictionary with export. Each feature follows the existing pattern of a `utils/` logic module paired with a numbered Streamlit page (or tab extension), with Supabase cloud storage gracefully falling back to local when unconfigured.

Watch for: user-supplied SQL passed unsandboxed to SQLite (confirmed); user-supplied regex compiled without timeout or length constraint (confirmed); template deletion on cloud is not verified against existence before returning success (confirmed).

## High-level view

The SQL interface hands raw user text to `pandasql.sqldf`, which creates a temporary SQLite database, loads the DataFrames into it, then executes the query string verbatim. Since this is a single-user analysis tool and SQLite is ephemeral, the blast radius is limited — but there is no guard against `ATTACH DATABASE` or `PRAGMA` commands that could touch the filesystem when deployed to Streamlit Cloud with shared access.

Column transforms are wired into the Data Cleaning page's existing undo system. The regex extract path accepts arbitrary user patterns and passes them directly to `pd.Series.str.extract` — a pathological pattern on a large text column could block the Streamlit worker thread.

The templates system mirrors the `(ok, payload)` tuple convention from `database.py` and splits routing between Supabase and local JSON files via `is_configured()`. Local file storage sanitizes names but the fallback search in `load_template` reads every JSON file in the directory on each call.

Annotations are session-state-only (not persisted), which means they vanish on refresh — a gap compared to the persistence story offered by templates and cloud workspace.

<summary>Issues (6)</summary>

1. **Unsandboxed SQL execution** — `pandasql.sqldf` passes user input to SQLite without restricting statements like `ATTACH DATABASE` or `PRAGMA`. Add a statement whitelist (SELECT only) or use `sqlite3.set_authorizer` to restrict capabilities. (confirmed)
2. **Unbounded regex in string_operations** — User-supplied regex is compiled and run against potentially millions of rows with no timeout or pattern-length limit. Cap pattern length and wrap in a timeout or use `re.compile` with flags to reject known-dangerous constructs. (confirmed)
3. **Template delete succeeds silently on cloud** — `delete_template` returns `(True, "Template deleted.")` even when the `eq("id", ...)` matched zero rows, because Supabase's delete doesn't raise on empty match. Check response data length before confirming. (confirmed)
4. **Local template listing reads all files per call** — `list_templates` in local mode opens and parses every JSON file in `.dataprism/templates/` on each invocation. `dp_templates_cache` is initialized in `data_loader.py` but never populated. (likely)
5. **Annotations are session-only, unlike templates** — Chart annotations stored in `st.session_state.dp_annotations` disappear on page refresh. Should be covered by the cloud save path or documented as ephemeral. (confirmed)
6. **Merge results overwrite each other** — Both join and concat save to `merged_df`, so a user cannot retain a join result and a concat result simultaneously. (confirmed)

## SQL execution without statement filtering

`run_sql_query` in `utils/sql_engine.py` passes the user's query string directly to `pandasql.sqldf`, which internally creates a temporary SQLite connection, loads each DataFrame as a table via `to_sql`, and runs the raw SQL.

```python
result_df = sqldf(query, local_env)
```

SQLite supports `ATTACH DATABASE '/tmp/exfil.db' AS exfil` and a range of PRAGMAs that can read/write files. In a Streamlit Cloud deployment accessible to "testers" (per the feedback-widget requirement), anyone with access can probe the filesystem. A lightweight mitigation: reject any query not beginning with `SELECT` (or `WITH`) after stripping whitespace and comments.

## Regex without bounds in column transforms

`string_operations` calls:

```python
extracted = df[column].astype(str).str.extract(f"({pattern})", expand=False)
```

The `pattern` variable comes directly from user input in `pages/3_Data_Cleaning.py`:

```python
re_pattern = st.text_input("Regex pattern:", ...)
```

No length limit, no pre-validation, no timeout. A pattern like `(a+)+$` on a column of long strings triggers catastrophic backtracking and hangs the Streamlit worker. Limit the pattern length (say 200 characters) and wrap the operation in a thread-based timeout.

## Template delete confirms success without verification

In `utils/templates.py`:

```python
client.table(T_TEMPLATES).delete().eq("id", template_id_or_name).execute()
return True, "Template deleted."
```

Supabase's PostgREST `DELETE` returns HTTP 200 with an empty `data` array when no rows match the filter. The function reports success regardless. Checking `len(resp.data) == 0` would let the caller know nothing was actually removed.

## Local template directory scan

`list_templates()` in local mode iterates every `.json` file in `.dataprism/templates/`, opens and parses each one, extracts metadata, and returns the list. The function is called on every page load of Advanced Analytics to populate the dropdown. `st.session_state.dp_templates_cache` is initialized in `data_loader.py` but never written to — wiring that up would eliminate repeated disk reads.

## Data dictionary descriptions not persisted

The Data Dictionary page stores user-written column descriptions in `st.session_state.dp_data_dictionary`, which persists only for the current browser session. There is no "Save to Cloud" button on this page despite the Supabase integration existing. The markdown/CSV export partially mitigates this (users can download their work), but re-importing is not supported.

## Merge result overwrite

Both the join path and the concatenation path store their output under the same session key (`merged_df`). A user performing a join then a concat loses the join result. Separate keys (e.g., `join_result_df` and `concat_result_df`) or a named-results pattern would preserve both.
